# Introduction

Data collected from multiple sources must be organized into a structured format before analysis. This process is known as **Data Modeling**.

A well-designed data model improves:

* Report performance
* Data consistency
* Ease of analysis
* Dashboard accuracy
* Business decision-making

Power BI uses a **semantic data model**, where tables are connected through relationships, enabling users to analyze data from different perspectives.

### Definition

> **Data Modeling** is the process of organizing, connecting, and structuring data into logical relationships so that it can be efficiently analyzed and reported.

---

# Why is Data Modeling Important?

Without proper data modeling:

* Reports become slow.
* Duplicate data increases.
* Calculations may be incorrect.
* Relationships become confusing.
* Maintenance becomes difficult.

### Benefits

* Faster report performance
* Better data integrity
* Reduced redundancy
* Easier DAX calculations
* Scalable business reports

# 1. Star Schema

## Definition

A **Star Schema** is a data model where a central **Fact Table** is connected directly to multiple **Dimension Tables**.

The structure resembles a star, with the fact table at the center and dimensions surrounding it.

---

## Structure

```text
                Product
                   │
                   │
Customer ─── Fact Sales ─── Date
                   │
                   │
               Store
```

---

## Example

### Fact Sales

| Date ID | Product ID | Customer ID | Store ID | Sales |
| ------- | ---------- | ----------- | -------- | ----: |
| 101     | P001       | C001        | S001     | 50000 |

### Product Dimension

| Product ID | Product Name | Category    |
| ---------- | ------------ | ----------- |
| P001       | Laptop       | Electronics |

### Customer Dimension

| Customer ID | Customer Name | City      |
| ----------- | ------------- | --------- |
| C001        | Ravi          | Bangalore |

### Date Dimension

| Date ID | Month   | Year |
| ------- | ------- | ---- |
| 101     | January | 2026 |

### Store Dimension

| Store ID | Store Name | Region |
| -------- | ---------- | ------ |
| S001     | MG Road    | South  |

---

## Advantages of Star Schema

* Simple design
* Faster query performance
* Easy to understand
* Ideal for Power BI
* Supports efficient reporting

---

## Disadvantages

* Some data redundancy
* Larger storage compared to Snowflake Schema

---

# 2. Snowflake Schema

## Definition

A **Snowflake Schema** is an extension of the Star Schema where dimension tables are further normalized into additional related tables.

It resembles a snowflake because dimensions branch into multiple levels.

---

## Structure

```text
                    Category
                        │
                        │
Product ────────────────┘
    │
    │
Fact Sales
    │
Customer
    │
City
    │
State
```

---

## Example

### Product Table

| Product ID | Product Name | Category ID |
| ---------- | ------------ | ----------- |
| P001       | Laptop       | CAT01       |

### Category Table

| Category ID | Category Name |
| ----------- | ------------- |
| CAT01       | Electronics   |

---

## Advantages

* Reduces data redundancy
* Better storage efficiency
* Easier maintenance of dimension data

---

## Disadvantages

* More complex relationships
* Slower queries due to additional joins
* More difficult for beginners

---

# Star Schema vs Snowflake Schema

| Feature               | Star Schema              | Snowflake Schema                 |
| --------------------- | ------------------------ | -------------------------------- |
| Structure             | Simple                   | More complex                     |
| Dimension Tables      | Denormalized             | Normalized                       |
| Performance           | Faster                   | Slightly slower                  |
| Storage               | More redundancy          | Less redundancy                  |
| Ease of Understanding | Easy                     | Moderate                         |
| Best Use              | Reporting and dashboards | Large enterprise data warehouses |

# 3. Fact and Dimension Tables

## A. Fact Table

### Definition

A **Fact Table** stores measurable business data (facts) such as sales, revenue, quantity, or profit.

It usually contains:

* Numeric values (measures)
* Foreign keys linking to dimension tables

---

### Characteristics

* Large number of rows
* Transactional data
* Used for calculations and KPIs

---

### Example

| Date ID | Product ID | Customer ID | Sales | Quantity |
| ------- | ---------- | ----------- | ----: | -------: |
| 101     | P001       | C001        | 50000 |        5 |
| 102     | P002       | C002        | 30000 |        3 |

---

## Examples of Fact Tables

* Sales
* Orders
* Transactions
* Attendance
* Inventory Movements

---

## B. Dimension Table

### Definition

A **Dimension Table** contains descriptive information that provides context to the facts.

---

### Characteristics

* Smaller tables
* Descriptive attributes
* Used for filtering, grouping, and slicing data

---

### Example

**Product Dimension**

| Product ID | Product Name | Category    | Brand |
| ---------- | ------------ | ----------- | ----- |
| P001       | Laptop       | Electronics | Dell  |

---

### Examples of Dimension Tables

* Customer
* Product
* Employee
* Store
* Date
* Region

---

# Fact Table vs Dimension Table

| Feature | Fact Table            | Dimension Table              |
| ------- | --------------------- | ---------------------------- |
| Stores  | Numeric measures      | Descriptive information      |
| Size    | Large                 | Small                        |
| Purpose | Business calculations | Categorization and filtering |
| Example | Sales Amount          | Product Name                 |

# 4. Relationships and Cardinality

## What is a Relationship?

A **relationship** connects two tables using a common column (key).

Example:

Customer ID in the **Customers** table is linked to Customer ID in the **Sales** table.

---

## Primary Key and Foreign Key

### Primary Key

A unique value identifying each record.

Example:

Customer ID

101

102

103

---

### Foreign Key

A column in another table referencing the primary key.

Example:

Sales Table

| Sale ID | Customer ID |
| ------- | ----------- |
| 5001    | 101         |

---

# Cardinality

**Cardinality** defines how records in one table relate to records in another table.

---

## Types of Cardinality

### 1. One-to-One (1:1)

Each record matches exactly one record.

Example:

Employee ↔ Passport

---

### 2. One-to-Many (1:*)

One record in one table relates to many records in another table.

This is the **most common relationship in Power BI**.

Example:

```text
Customer
   │
   ├── Order 1
   ├── Order 2
   └── Order 3
```

One customer can place many orders.

---

### 3. Many-to-One (*:1)

The reverse of One-to-Many.

Many sales records relate to one product.

---

### 4. Many-to-Many (*:*)

Multiple records relate to multiple records.

Example:

Students ↔ Courses

One student studies many courses, and one course has many students.

---

# Cross Filter Direction

Power BI uses filter propagation between related tables.

## Single Direction

Filters move in one direction only.

Recommended for most models.

---

## Both Directions

Filters move in both directions.

Useful in some advanced scenarios but should be used carefully to avoid ambiguity.

---

# Best Practices for Relationships

* Use unique primary keys.
* Keep relationships active whenever possible.
* Avoid unnecessary many-to-many relationships.
* Ensure matching data types.
* Prefer a **Star Schema** for Power BI models.

# 5. Stakeholder Collaboration in Data Modeling

## Who is a Stakeholder?

A **stakeholder** is any individual or group involved in defining, creating, using, or maintaining the data model.

Successful data modeling requires collaboration among technical teams and business users.

---

## Types of Stakeholders

| Stakeholder                    | Role                                            |
| ------------------------------ | ----------------------------------------------- |
| Business Users                 | Define reporting requirements and KPIs          |
| Business Analysts              | Translate business needs into data requirements |
| Data Analysts                  | Create reports and dashboards                   |
| Data Engineers                 | Prepare and integrate data sources              |
| Database Administrators (DBAs) | Manage databases and optimize performance       |
| Power BI Developers            | Build the data model and reports                |
| Project Managers               | Coordinate timelines and deliverables           |
| Senior Management              | Use dashboards for strategic decisions          |

---

## Why Collaboration is Important

Stakeholder collaboration helps to:

* Understand business objectives.
* Identify key performance indicators (KPIs).
* Ensure correct business definitions.
* Validate data quality.
* Improve report usability.
* Reduce errors and rework.

---

## Example: Retail Company

A retail company wants a Sales Dashboard.

### Business Manager

Requests:

* Monthly Sales
* Top Products
* Regional Performance

### Finance Team

Requests:

* Profit Margin
* Revenue
* Expenses

### Inventory Team

Requests:

* Stock Levels
* Reorder Status

### Power BI Developer

Creates:

* Fact Sales table
* Product, Customer, Date, and Store dimensions
* Relationships between tables
* Dashboards meeting all stakeholder requirements

# Real-World Case Study

### Problem

Sales data existed in SQL Server, customer details in Excel, and product information in SAP.

### Result

* Faster dashboard performance.
* Consistent business metrics.
* Better inventory and sales analysis.
* Improved decision-making across departments.

# Classroom Practical (2 Hours)

Students will complete the following activities:

1. Import **Sales**, **Customer**, **Product**, and **Date** tables into Power BI.
2. Design a **Star Schema** by placing the **Sales** table at the center and connecting it to the dimension tables.
3. Create **One-to-Many** relationships using **Customer ID**, **Product ID**, and **Date ID**.
4. Observe how filters propagate across related tables in report visualizations.
5. Compare the Star Schema with a simple Snowflake Schema by creating a separate **Category** table linked to the **Product** table.
6. Discuss with classmates acting as different stakeholders (Sales Manager, Finance Manager, Inventory Manager) to identify the KPIs each would require and how the data model supports those requirements.
